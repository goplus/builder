# 设计审查清单

> - 文档类型：Checklist / 设计表达质量检查
> - 稳定性：随设计系统和产品复杂度微调
> - 何时阅读：提交 Design PR 或 Demo PR 前，或评审设计表达质量时
> - 维护原则：本文检查「设计表达质量」，不检查 Pencil 文件组织、资源路径和资产入库质量。资产入库检查见 [设计资产审查清单](./design-asset-review-checklist.zh.md)。

## 使用范围

提交 PR 前，作者先完成自查。评审者评审时，可按相同清单逐项确认。

不是所有条目都必须适用于每个需求。对于不适用的项，应在 PR 中说明「不涉及」。

## 本文不检查什么

本文不检查：

- Pencil 文件组织；
- 资源路径和引用关系；
- 文件、节点和资源命名；
- Git 状态和暂存范围。

这些内容使用 [设计资产审查清单](./design-asset-review-checklist.zh.md) 检查。

## 检查清单

### 1. 问题与目标

- [ ] 是否能从 Issue 中看出本次设计要解决什么问题？
- [ ] 设计目标是否与 Issue 一致？
- [ ] 是否避免只描述方案，而没有说明用户问题？
- [ ] 是否说明本次设计的成功标准或验收方向？
- [ ] 是否说明当前设计属于新增、优化、修复还是探索？

### 2. 用户路径

- [ ] 是否覆盖用户从进入页面到完成目标的主要路径？
- [ ] 是否说明用户如何触发关键操作？
- [ ] 是否说明操作后的反馈？
- [ ] 是否说明用户如何退出、取消、返回或恢复？
- [ ] 是否考虑用户误操作后的处理方式？

### 3. 页面结构与信息层级

- [ ] 页面主次信息是否清晰？
- [ ] 主要操作是否足够突出？
- [ ] 次要操作是否不会干扰主任务？
- [ ] 页面区域划分是否符合现有产品结构？
- [ ] 信息密度是否合理？
- [ ] 是否避免重复入口或重复表达？

### 4. 组件与设计系统

- [ ] 是否优先复用现有 Design System 组件？
- [ ] 是否使用已有 Design Token？
- [ ] 是否出现未定义的新样式？
- [ ] 若新增组件，是否说明新增原因和复用范围？
- [ ] 若修改组件，是否说明影响范围？
- [ ] 是否避免在页面里局部复制组件系统能力？

### 5. 状态覆盖

- [ ] 是否定义默认状态？
- [ ] 是否定义 `hover`、`focus`、`active` 等交互状态？
- [ ] 是否定义 `selected` 或 `current` 状态？
- [ ] 是否定义 `disabled` 状态？
- [ ] 是否定义 `loading` 状态？
- [ ] 是否定义 `empty` 状态？
- [ ] 是否定义 `error` 状态？
- [ ] 是否定义 `success` 或完成状态？
- [ ] 若涉及 AI 生成，是否定义 `generating`、`retry`、`cancel`、`failed` 等状态？
- [ ] 若涉及展开收起，是否定义 `collapsed` 与 `expanded` 状态？

### 6. 交互表达

- [ ] 点击、拖拽、`hover`、键盘操作等触发方式是否清楚？
- [ ] 页面跳转逻辑是否清楚？
- [ ] 弹窗、浮层、Tooltip、Popover 的出现和消失规则是否清楚？
- [ ] 面板展开、收起、避让、吸附等行为是否清楚？
- [ ] 是否说明必要动画或过渡？
- [ ] 是否说明操作不可用时的反馈？
- [ ] 是否说明静态稿无法表达、需要在 Demo PR 中重点验证的交互？

### 7. 异常与边界场景

- [ ] 是否考虑无数据场景？
- [ ] 是否考虑加载失败场景？
- [ ] 是否考虑权限不足或额度不足场景？
- [ ] 是否考虑网络异常或生成失败场景？
- [ ] 是否考虑内容过长、名称过长、图片加载失败等边界？
- [ ] 是否考虑用户连续点击、重复触发或中断操作？

### 8. 视觉一致性

- [ ] 是否与当前产品既有页面风格一致？
- [ ] 间距、圆角、阴影、边框是否统一？
- [ ] 字号、字重、行高是否符合现有规范？
- [ ] 图标尺寸和视觉重量是否一致？
- [ ] 卡片、列表、面板、按钮等组件密度是否一致？
- [ ] 是否避免过度装饰或只服务视觉、不服务任务的元素？

### 9. 响应式与适配

- [ ] 是否考虑不同屏幕宽度？
- [ ] 是否考虑编辑器区域缩放或面板尺寸变化？
- [ ] 是否考虑文本换行、截断和 Tooltip 补充？
- [ ] 是否考虑列表数量较多时的滚动与固定区域？
- [ ] 是否考虑移动端或窄屏场景；若不涉及，是否说明不涉及？

### 10. 可访问性与可理解性

- [ ] 颜色是否有足够区分度？
- [ ] 是否避免只依赖颜色表达状态？
- [ ] 交互目标是否足够可点击？
- [ ] 禁用、错误、成功等状态是否有文字或图标辅助？
- [ ] 重要操作是否有清晰标签？
- [ ] 是否避免让用户必须猜测系统正在做什么？

### 11. 文案与标点

- [ ] 中文环境是否使用全角标点。
- [ ] 是否使用中文全角书名号「xxx」表达引用或强调对象。
- [ ] 中文省略号是否使用「……」，而不是英文省略号「...」。
- [ ] 中英文混排是否保留必要空格，例如「录制的 Demo 视频」。
- [ ] 日期和时间格式是否清晰，例如「2026 年 06 月 04 日」。
- [ ] 标题、按钮、提示语的语气是否一致？
- [ ] 错误提示是否说明问题和下一步？
- [ ] `loading` 或生成中提示是否说明系统正在做什么？

### 12. Demo PR 验证范围

- [ ] Demo PR 是否说明验证范围？
- [ ] 是否存在需要重点验证的动态交互？
- [ ] 是否说明 Demo PR 的验证范围与设计风险是否匹配？
- [ ] Demo 证据是否足以支持团队形成设计共识？
- [ ] Demo PR 是否保持 Draft？

### 13. 决策记录

- [ ] 是否记录关键设计决策？
- [ ] 是否说明与旧方案的差异？
- [ ] 是否说明被放弃方案的原因；若不重要，是否已清理？
- [ ] 是否说明后续生产实现需要承接的内容？
- [ ] 是否存在仍需产品、设计或研发确认的问题？

## 评审结论

评审者可以按下面方式给出结论：

```text
Review result: Approved
- 设计目标与 Issue 一致。
- 关键状态已覆盖。
- Demo PR 的验证范围与设计风险匹配。
```

或：

```text
Review result: Changes requested
- 需要补充 `error` 状态。
- 需要说明 `Tooltip` 消失规则。
- 需要补充后续生产实现承接点。
```
