# 设计到 Demo 工作流（Design to Demo Workflow）

本文档定义在设计资产（当前仓库为 Pencil 文件）中完成设计改动后，经 Design PR、Demo PR 验证，再交给后续生产实现承接的协作流程。

## 定义与边界

设计到 Demo 工作流用于在后续生产实现前验证设计意图，重点覆盖关键交互、页面状态和设计到代码的初步翻译。

本文适用于需要在设计资产（当前仓库为 Pencil 文件）中完成设计改动，并通过前端 demo 验证体验的工作；不适用于只修改生产代码、不涉及设计资产的工程改动，或只更新文档、配置、后端逻辑的任务。

本文不定义设计资产（当前仓库为 Pencil 文件）的文件规范、后续生产实现规范或后端实现规则；相关内容见本文末尾的文档关系。

## 核心对象

在当前仓库中，核心对象的目标分支和合并规则如下：

| 对象 | 定义 | 目标分支 | 合并规则 |
| ---- | ---- | -------- | -------- |
| Issue | 描述用户问题、设计目标、背景材料和验收方向。 | 不适用 | 不承载代码或设计资产。 |
| Design PR | 提交设计资产改动，说明设计意图、变更范围和设计自查结果。 | `ui` | 合并前只保留应进入仓库历史的设计资产。 |
| Demo PR | 提交前端 demo 代码和预览上下文，用于体验关键交互和页面状态。 | `ui` | 必须保持 draft 状态，避免 demo 代码被误合并；作为验证与交接上下文保留。 |

后续生产实现可以用 Code PR 表达，用于基于已验证的设计意图继续推进；但 Code PR 不属于本文定义的主要环节，其模板、测试和合并规则不在本文档中界定。

同一个需求应通过同一个 issue 串联 Design PR 和 Demo PR，并保留后续生产实现需要承接的信息；Design PR 和 Demo PR 职责不同，不能互相替代。

## 前置条件

开始设计工作前应满足：

- 已有明确 issue，且 issue 描述了背景、目标和验收方向。
- 已理解 `dev` 分支用于真实开发，`ui` 分支用于设计验证。
- 当前工作包含需要进入仓库历史的设计资产。
- 如需提交 Demo PR，issue 应明确 demo 要验证的入口、交互、状态或用户路径。

如果体验过程中发现新问题，应先创建 issue，再基于该 issue 进入本流程。

## 责任边界

| 角色 | 责任 |
| ---- | ---- |
| 设计研发协作者 | 维护设计资产（当前仓库为 Pencil 文件），提交 Design PR，确认设计意图和关键状态。 |
| Demo 实现者 | 基于设计改动生成前端 demo，提交 Demo PR，并说明 demo 覆盖范围和非生产代码边界。 |
| 承接开发人员 | 查看 issue、Design PR、Demo PR 和预览环境，确认后续生产实现需要承接的交互、状态和工程要求。 |
| Review 参与者 | 在 Design PR 和 Demo PR 中确认设计表达、交互体验、预览结果和后续交接信息是否充分。 |

同一人可以承担多个角色，但 PR 描述中必须让责任和交付物边界清晰可追溯。

## 交付规则

Design PR 应包含：

- 设计资产改动。
- 必要的说明、截图或录屏。
- 关联 issue。
- 设计自查结果或关键验证说明。

Demo PR 应包含：

- 前端 demo 代码。
- 可体验的预览环境或验证说明。
- 与 issue 直接相关的交互和状态说明。
- 后续生产实现需要承接的内容。

Demo PR 必须保持 draft 状态，默认不合并。如果其中包含需要进入仓库历史的设计资产改动，应先拆分设计资产改动和 demo 代码改动。

demo 需要表达：

- 关键交互。
- 页面状态变化。
- 与 issue 直接相关的用户路径。
- 设计到代码的初步翻译结果。

demo 不需要表达：

- 完整生产业务逻辑。
- 真实权限、持久化和异常兜底。
- 生产级代码结构和测试覆盖。

demo 可以基于现有前端代码保留现有页面和功能，但本次改动和验证范围应只覆盖当前 issue 相关内容。

## 工作流程

### 1. 基于 issue 开始工作

设计工作必须关联 issue。

开始前应确认：

- issue 是否描述了用户问题或设计目标。
- issue 是否已有相关截图、设计稿、讨论记录或历史 PR。
- 当前工作是新设计、设计调整，还是对已有体验问题的修复。

### 2. 同步最新前端基础

在当前仓库中，开始设计验证前应先让 `ui` 分支接近最新 `dev` 分支状态。

同步 `dev` 到 `ui` 应通过独立 PR 完成。该 PR 只承担同步基础代码的职责，不应混入具体设计改动。

该同步通常由熟悉 Git 的协作者或维护者执行；设计研发协作者开始具体设计工作前，只需要确认当前 issue 基于最新可用的 `ui` 分支。

### 3. 在设计资产中完成设计改动

设计改动应优先进入设计资产（当前仓库为 Pencil 文件）。

设计资产改动应表达：

- 页面结构。
- 组件状态。
- 关键交互路径。
- 与当前 issue 相关的设计决策。

Pencil 文件规范见：[Pencil 文件规范（Pencil Guidelines）](./pencil-guidelines.zh.md)。

### 4. 执行设计自查

提交 Design PR 前应完成设计自查。

自查内容包括：

- 设计状态是否完整。
- 交互是否明确。
- 是否复用已有组件和设计资产。
- 设计资产是否符合文件规范。

参考文档：

- [设计审查清单（Design Review Checklist）](./design-review-checklist.md)
- [设计资产校验（Design Asset Validation）](./design-asset-validation.md)

### 5. 提交 Design PR

Design PR 应基于 `ui` 分支，并关联原 issue。它只提交设计资产改动；需要 demo 验证时，另行提交 Demo PR。

### 6. 提交 Demo PR

当设计资产（当前仓库为 Pencil 文件）无法充分表达交互体验时，应基于设计改动生成前端 demo，并提交关联同一个 issue、指向 `ui` 分支的 Demo PR。

### 7. 使用 PR 预览环境验收

PR 预览环境用于体验 demo。设计研发协作者和后续承接开发人员都应查看预览环境。

验收时应关注：

- 关键路径是否可进入。
- 交互是否符合设计意图。
- 页面状态是否覆盖当前 issue 的主要场景。
- 是否存在需要回到设计资产修改的设计问题。
- 是否存在需要由后续生产实现承接的工程问题。

预览通过标准是：当前 issue 的关键交互和主要状态可以被体验，且没有阻碍开发人员理解设计意图的问题。

如果预览暴露的是当前设计表达问题，应回到设计资产修改，并更新 Design PR 或 Demo PR。如果预览暴露的是新的产品问题或独立体验问题，应创建新 issue，并基于新 issue 重新进入本流程。

### 8. 合并前清理 demo 代码

如果 Design PR 中临时包含 demo 代码，合并前应按交付规则删除不预期合并的前端 demo 代码，只保留应进入仓库历史的设计资产改动。

清理后应确认：

- PR 中仍保留设计资产改动。
- PR 中不再包含仅用于 demo 的前端代码。
- PR 仍然关联原 issue。
- Demo PR、预览环境、截图、录屏或说明文档仍能支撑后续生产实现理解设计意图。

### 9. 交接给后续生产实现

后续生产实现可以用 Code PR 表达，基于已验证的设计意图继续推进。本文只要求 Design PR 和 Demo PR 保留足够的交接信息，不定义 Code PR 的模板、测试要求或合并规则。

后续生产实现需要从本流程中承接：

- 已验证的设计意图。
- 可迁移的结构、样式和交互表达。
- 需要生产实现处理的真实数据、权限、状态管理和异常处理。

完成 Design PR 和 Demo PR 验证后，应将后续生产实现 assignee 给对应开发人员。Assignee 应能从 issue、Design PR、Demo PR 和预览环境中理解设计目标与关键交互。
