# Pencil 设计资产规范

> 文档类型：Tool Guide / 当前工具实践
> 稳定性：随 Pencil 和设计工具链更新
> 适用对象：维护 `.pen`、`.lib.pen`、`ui/images/*` 的产品设计工程化协作者
> 何时阅读：修改 Pencil 文件、从 Figma 搬运设计、提交 Design PR 前
> 维护原则：Pencil 是当前阶段的设计资产载体，不是这套工作流的核心。未来工具替换时，应保留资产版本化、可审查、可追溯、可交接的原则。

## 定位

本文中的设计资产，特指采用 Pencil 的项目仓库中预期进入版本历史，并作为设计决策、页面结构、组件状态或工程交接依据的 Pencil 文件及其依赖资源。XBuilder 是当前阶段的主要试验场。

当前阶段，Pencil 的作用是把设计从个人工具中的视觉稿，转化为项目仓库中可管理、可审查、可追溯的协作资产。

Pencil 不是这套工作流的核心。它只是当前阶段的设计资产载体。

## 资产类型

| 类型 | 示例 | 作用 |
| ---- | ---- | ---- |
| 页面级文件 | `ui/pages/spx/*.pen` | 表达页面结构、页面状态、业务场景和关键交互说明。 |
| 组件库文件 | `ui/components/spx/*.lib.pen` | 维护可复用组件、组件状态、变量和 Slot。 |
| 依赖资源 | `ui/images/*` | 存放 `.pen` 文件引用的图片、字体、图标等资源。 |
| 规范文档 | `ui/docs/*.md` | 说明工作流、命名、评审和提交规则。 |

## 文件职责

### 页面级文件

页面级文件用于表达当前 Issue 需要审查和交接的页面结构、关键布局与主要状态。

页面级文件应包含：

- 当前页面的稳定结构；
- 关键用户路径；
- 主要组件使用方式；
- 必要的页面状态；
- 与当前 Issue 相关的交互说明；
- 后续生产实现需要理解的视觉或结构信息。

页面级文件不应长期保存：

- 废弃探索稿；
- 临时截图；
- 无关页面；
- 无法追溯来源的图片；
- 只用于个人对照的 Figma 搬运中间状态。

### 组件库文件

组件库文件是设计系统的核心来源。

页面文件应优先引用组件库中的组件，而不是复制组件、复制样式或自行维护一套局部组件。

组件库文件应维护：

- 可复用组件；
- 组件状态；
- 可替换 Slot；
- 设计变量；
- 稳定的组件命名；
- 对页面可复用的结构。

修改组件库文件时，应特别谨慎，因为它可能影响多个页面。

### 依赖资源

`.pen` 文件引用的图片、字体、图标等资源应放在仓库约定目录中，当前建议放在：

```text
ui/images/
```

不要依赖个人本地路径。其他协作者拉取仓库后，应能正常打开设计资产。

## 页面文件与组件库文件的关系

简单原则：

```text
组件库负责稳定组件和状态。
页面文件负责表达具体业务场景。
```

页面文件不应复制组件库中的稳定组件。如果页面需要一个新组件，应先判断：

- 是否已有组件可复用；
- 是否只是当前页面的局部结构；
- 是否会被多个页面复用；
- 是否应该进入组件库；
- 是否会影响已有页面。

如果需要新增或修改组件库，应在 PR 中说明影响范围。

## Slot 使用原则

Slot 用于表达组件中的可替换内容位。

页面级文件应通过 Slot 填充页面内容，而不是直接拆开组件内部结构进行改写。

建议：

- 组件根节点保持稳定；
- 可替换区域使用 `Slot/<Role>` 命名；
- 页面内容填入 Slot；
- 不在页面里复制一份组件内部结构；
- 如果 Slot 无法满足页面需求，应回到组件库判断是否需要扩展组件。

命名规则见 [设计资产命名](./design-asset-naming.zh.md)。

## 结构化布局

Pencil 文件应尽量表达结构关系，而不是保存一次性的视觉快照。

建议：

- 使用容器表达布局；
- 使用间距、对齐和层级表达结构；
- 保持页面区域命名清晰；
- 让节点层级能够反映真实页面结构；
- 避免用大量绝对位置固化临时效果；
- 避免无意义分组。

结构越清晰，后续评审、Demo 和生产实现越容易理解。

## 状态表达

影响用户理解、交互路径、前端实现或验收结果的状态，应进入设计资产。

常见状态包括：

- `default`；
- `hover`；
- `focus`；
- `active`；
- `selected`；
- `disabled`；
- `loading`；
- `empty`；
- `error`；
- `success`；
- `generating`；
- `retry`；
- `collapsed`；
- `expanded`。

不需要把所有可能状态机械地画出来，但必须覆盖当前 Issue 的关键路径。

如果某个状态只通过静态稿难以表达，应进入 Demo 验证。

## 从 Figma 搬运到 Pencil

本节只说明迁移原则。具体迁移步骤、变量映射、Slot 使用、跨文件引用、图片/Icon 处理和常见异常排查，见 [Figma 到 Pencil 迁移指南](./figma-to-pencil-migration.zh.md)。

从 Figma 搬运到 Pencil 时，不应原样复制。

Figma 中可能包含大量探索图层、临时分组、废弃方案、截图、标注、设计过程记录。进入 Pencil 前，需要整理成可维护、可审查、可交接的设计资产。

推荐处理方式：

- 保留设计意图；
- 清理临时图层；
- 删除废弃方案；
- 使用结构化布局；
- 复用组件库；
- 整理关键状态；
- 删除与当前 Issue 无关的探索内容；
- 在 PR 中说明与原方案的差异和待确认点。

如果 Figma 方案与当前组件库不一致，应先判断差异是否代表新的设计决策。不能确认时，应在 PR 中说明待确认点。


## 什么内容应该进入仓库历史

应该进入仓库历史的内容：

- 当前 Issue 相关页面结构；
- 关键组件状态；
- 影响前端实现的设计决策；
- 后续实现需要依据的资源；
- 与设计系统相关的组件更新；
- 需要被评审的页面和组件变化。

不应该进入仓库历史的内容：

- 个人草稿；
- 临时截图；
- 废弃方案；
- `copy`、`final`、`latest` 等临时文件；
- 无法追溯来源的图片；
- 与当前 Issue 无关的 Figma 搬运内容；
- `.DS_Store`、`__MACOSX/` 等系统文件。

## Pencil 与 Git 的配合

修改 Pencil 文件后，不要只依赖本地保存。应通过 Git 形成可恢复版本。

推荐流程：

```bash
# 查看当前文件变化
git status --short

# 确认具体变化
git diff --stat

# 添加相关文件
git add ui/pages/spx/<page>.pen
git add ui/components/spx/<component>.lib.pen
git add ui/images/<resource>

# 提交
git commit -m "design: update <scope>"
```

如果一直在修改同一个已被 Git 追踪的 `.lib.pen` 文件，可以使用：

```bash
git commit -am "design: update component library"
```

但注意：`git commit -am` 不会提交新文件。新增图片、新增 `.pen` 文件时，必须先 `git add <file>`。

## 防丢失经验

Pencil 文件可能因为本地路径、缓存、应用异常或误操作出现丢失风险。建议：

- 关键节点及时保存；
- 完成一个明确小阶段后提交 Commit；
- 不要只执行 `git add`，要执行 `git commit`；
- 每次提交前检查是否包含相关图片和资源；
- 修改 `.lib.pen` 前先确认当前分支干净；
- 大范围组件库修改前先创建独立分支；
- 不要长期在一个巨大未提交状态中连续工作。

更安全的节奏是：

```text
完成一个可说明的小变化
  ↓
检查文件范围
  ↓
commit
  ↓
继续下一步
```

## Design PR 中需要说明的内容

修改 Pencil 设计资产时，Design PR 至少说明：

- 修改了哪些 `.pen` 或 `.lib.pen` 文件；
- 是否新增或修改了 `ui/images/*` 资源；
- 是否影响组件库；
- 是否影响已有页面；
- 本次设计覆盖了哪些状态；
- Demo PR 的验证范围；
- 是否存在待确认点。

提交前使用 [设计资产审查清单](./design-asset-review-checklist.zh.md) 自查。

## 未来工具替换原则

未来如果 Pencil 被替换，新工具至少应继续支持以下工作流要求：

- 设计资产可以进入版本管理；
- 设计变更可以被评审；
- 设计文件能被多人协作理解；
- 页面结构、组件状态和资源引用可追溯；
- 关键交互可以进入 Demo 验证；
- 后续生产实现能从设计资产中获得清晰依据。

如果某个新工具不能支撑这些能力，就不能直接替代当前工作流中的 Pencil 角色。
