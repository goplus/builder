# Pencil 设计资产规范（Pencil Design Asset Guidelines）

本文中的设计资产，特指 `ui/` 目录中预期进入仓库历史，并作为设计决策、页面结构、组件状态或工程交接依据的 Pencil 文件；下文简称设计资产。

本文档定义设计资产的文件位置、组织方式、引用关系和迁移规则。

范围上，本文不定义 issue 到 demo 的操作步骤、不定义真实前端实现规则，也不替代设计审查清单和设计资产校验文档。

## 定义与边界

除设计资产外，本文档还使用以下术语：

- 页面级文件：用于表达单个页面、页面状态或业务场景的 `.pen` 文件。
- 组件库文件：用于维护可复用组件、组件状态和 Pencil 变量的 `.lib.pen` 文件。
- Pencil 变量：在 Pencil 文件中维护，并供组件或页面引用的颜色、字体、间距、圆角、阴影等设计值。
- Slot 节点：组件库文件中用于声明可替换内容位置的 Pencil 节点。页面级文件应通过 Slot 节点填充页面内容，而不是复制或改写组件库内部结构。
- 结构化布局：使用容器、间距、对齐、约束和 Slot 节点表达页面结构关系，而不是用固定宽高或绝对位置保存一次性的视觉结果。
- 临时参考：只用于讨论、对照或迁移过程的截图、探索稿、标注图和外部资源。

设计资产应表达稳定内容，而不是保存完整探索过程。探索内容只有在能说明设计决策、状态覆盖或工程交接要求时，才应进入对应的页面级文件或组件库文件。

## 资产位置

当前仓库中，设计资产及其依赖资源按以下路径维护：

| 路径 | 职责 |
| ---- | ---- |
| `ui/components/spx/builder-component.lib.pen` | 设计资产；组件库文件，维护可复用组件、组件状态和 Pencil 变量。 |
| `ui/pages/spx/*.pen` | 设计资产；页面级文件，维护页面结构、页面状态和业务场景设计。 |
| `ui/images/*` | 依赖资源；维护被 `.pen` 文件引用的字体、图片和其他设计资源。 |

新增页面级文件应放在对应产品域目录下。当前 SPX/XBuilder 页面应放在 `ui/pages/spx/`。

组件、节点、变量、图片和文档的具体命名规则见 [设计资产命名（Design Asset Naming）](./design-asset-naming.md)。本文只约束设计资产的职责边界。

## 文件命名

Pencil 文件名应表达页面、组件库或业务对象，不应表达临时状态。

文件名不应包含：

- `new file`
- `copy`
- `test`
- `final`
- `latest`
- `done`
- `approved`

组件体系的 SSOT（Single Source of Truth）是 `ui/components/spx/builder-component.lib.pen`。页面级文件不应另建组件来源；如需新增或修改组件，应进入组件库文件。

需要进入仓库历史的设计资产，必须能通过 issue、Design PR 或相关说明追溯其用途。

## 页面、组件和状态组织

页面级文件用于表达当前 issue 需要审查和交接的页面结构、关键布局与主要状态。必要的交互说明或状态标注可使用 Pencil 自带的 Note 功能记录。

组件库文件用于维护可复用组件、组件状态和 Pencil 变量。组件命名与层级应对应真实前端组件语义，而不是表达临时图层结构。

状态进入设计资产的判断标准是：影响用户理解、交互路径、前端实现或验收结果的状态应被表达；仅用于视觉探索或未采用方案的状态不应作为正式设计资产长期保留。

## 引用规则

页面级文件应通过组件库文件中配置好的 Slot 节点使用组件，不应复制组件库中的稳定组件、字体或图标资源。确实需要新增或修改组件时，应在 PR 中说明原因和使用范围。

设计资产引用的依赖资源应满足：

- 位于 `ui/images/` 或当前仓库约定的设计资源目录。
- 资源能被相关 `.pen` 文件追溯引用。
- 资源来源、用途和保留原因能在 PR 中说明。
- 不依赖个人本地文件、未提交资源或无法复现的外部状态。

设计资产在协作链路中的角色是记录设计稿和设计决策。Issue、Design PR、Demo PR 和 Code PR 的完整关系见 [设计到 Demo 工作流（Design to Demo Workflow）](./design-to-demo-workflow.zh.md)。

## 从 Figma 搬运到 Pencil

从 Figma 搬运到 Pencil 时，应将 Figma 内容整理为可维护、可实现的设计资产，而不是原样复制临时结构。页面结构应采用结构化布局表达，避免用固定宽高或绝对位置固化临时视觉结果。

迁移时应保留设计意图，清理临时图层、草稿层、重复对象和无关页面；可复用部分应整理为组件库组件，页面内容应通过组件库文件中配置好的 Slot 节点使用组件。

截图、临时命名、实验分组和与当前 issue 无关的探索内容，不应作为可编辑设计资产保留。

如果 Figma 与当前组件库存在差异，应优先明确差异是否代表新的设计决策。不能确认时，应在 PR 中说明待确认点。

## 进入仓库历史的标准

与当前 issue 相关的页面结构、组件状态、交互状态变化和影响前端实现的设计决策，应进入对应的页面级文件或组件库文件。

以下内容可以作为临时参考，但不应作为正式设计资产长期保留：

- Figma 原始截图。
- 没有整理过的探索稿。
- 与当前 issue 无关的备选方案。
- 只用于沟通的标注图。
- 无法追溯来源的图片、字体或图标资源。

如果临时参考内容必须随 PR 保留，应在 PR 描述中说明原因和后续清理计划。

## 审查口径

设计资产应满足：

- 可审查：PR diff、截图、录屏或说明能让 reviewer 理解设计资产发生了什么变化。
- 可追溯：设计改动能关联到 issue、Design PR 或明确的设计决策。
- 可复用：稳定组件、状态和变量能被后续页面或组件继续使用。
- 可交接：后续 Demo PR 或 Code PR 能从设计资产中理解页面结构、状态和关键交互。

如果某个设计资产改动无法满足上述任一条件，应在 PR 中说明原因、风险和后续处理方式。

提交前应参考：

- [设计审查清单（Design Review Checklist）](./design-review-checklist.md)
- [设计资产校验（Design Asset Validation）](./design-asset-validation.md)
