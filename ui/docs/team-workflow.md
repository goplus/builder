# 团队工作流

## 概述

本文档定义 Builder 设计与研发协作流程，目标是在开发实现前完成设计意图、交互体验和代码表达之间的对齐。

核心方向是 **设计左移**：让设计工作更早进入工程流程，通过可审查、可体验、可追溯的交付物降低后续实现偏差。

## 背景

当前团队正在梳理设计研发工作流。设计交付不应只停留在静态稿，而应尽量在进入真实前端实现前完成以下工作：

- 明确设计意图。
- 验证关键交互。
- 将设计稿翻译成可运行或可参考的代码表达。

这样可以让设计问题更早暴露，也能让前端工程师在实现阶段减少重复理解和重新翻译设计稿的成本。

## Demo 的价值

Demo 是设计稿到真实前端实现之间的过渡交付物。它的主要价值是：

- 体现交互，而不只是展示静态视觉结果。
- 在开发实现之前，就能提前体验关键流程和状态变化。
- 完成从设计稿到代码的初步翻译，减少前端工程师后续实现时的工作量。

Demo 不等同于生产代码。它用于表达设计意图、验证体验，并为真实前端实现提供更明确的参考。

## 两种交付形式

| 交付形式 | 工作基础 | 产出 | 是否合并 |
| ------- | ------- | ---- | -------- |
| 生产实现 | 基于 `dev` 分支工作 | 真实前端代码 | 会被合并 |
| 设计 Demo | 基于 `ui` 分支工作 | 可体验的 demo 代码 | 前端代码改动不预期合并 |

### 基于 `dev` 分支

当需求进入真实开发阶段时，应基于 `dev` 分支工作。此类 PR 承载生产实现，代码会进入真实前端工程，并最终合并。

### 基于 `ui` 分支

当需求处于设计验证阶段时，应基于 `ui` 分支工作。此类 PR 用于产出 demo，让团队提前体验设计意图和交互效果。

`ui` 分支中的前端代码改动主要服务于设计验证，不应默认视为最终生产实现。

## 预期工作流

未来设计工作将逐步迁移到 Pencil。

团队应定期将 `dev` 分支的最新代码同步到 `ui` 分支，确保设计 demo 所依赖的前端基础尽量接近当前真实工程状态。

设计 PR 中通常会包含两类改动：

- Pencil 文件改动：用于记录设计稿、组件状态和设计决策。
- 前端代码改动：用于生成 demo，表达设计意图和关键交互。

需要注意的是，设计 PR 中对前端代码的改动不预期被合并；Pencil 文件的改动才是需要合并的设计资产。

因此，每次设计 PR 合并前，需要删除前端代码改动，只保留应进入仓库历史的 Pencil 文件改动。

## 责任边界

### 设计阶段

设计阶段负责回答：

- 设计目标是什么？
- 关键交互如何体验？
- 页面、组件和状态应该如何呈现？
- Demo 是否足够表达设计意图？

### 开发阶段

开发阶段负责回答：

- 真实前端如何实现该设计？
- 哪些 demo 中的表达可以迁移到生产代码？
- 哪些逻辑、数据、权限和异常状态需要在真实工程中补齐？
- 最终实现是否满足需求验收标准？
