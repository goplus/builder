# 产品设计工程化协作准入准备

> 文档类型：Onboarding / 准入标准
> 稳定性：长期稳定，随团队实践微调
> 适用对象：首次参与产品设计工程化工作流的产品、设计、内容、运营、项目管理协作者、AI Agent 使用者
> 何时阅读：参与 Issue、Design PR 或 Demo PR 之前
> 维护原则：本文只定义加入工作流前需要理解的概念、具备的基础能力和准入完成标准；阅读路径由 README 维护，工具细节由专题文档维护。

## 目的

本文档用于判断协作者是否具备参与产品设计工程化工作流的最低协作能力。

协作者不需要成为前端工程师，也不需要独立完成生产代码实现。但协作者必须理解：进入仓库的设计资产、Demo 验证材料、PR 描述和评审记录，都是项目协作资产，而不是个人草稿或临时说明。

## 最低概念要求

| 概念 | 需要理解到什么程度 |
| ---- | ------------------ |
| Repository | 知道项目文件、设计资产、文档和 Demo 都在版本库中管理。 |
| Issue | 知道 Issue 是工作上下文的起点，用来描述问题、目标和验收方向。 |
| Branch | 知道分支用于隔离不同工作线，不直接在公共目标分支上修改。 |
| Commit | 知道 Commit 是一次可追溯的改动记录，不是随手保存。 |
| Remote | 知道本地改动需要通过远端分支进入协作流程。 |
| Pull Request | 知道 PR 是提交改动、说明范围、接受评审和保留讨论记录的入口。 |
| Review | 知道评审用于确认设计表达、资产组织、Demo 验证和交接信息是否充分。 |
| Assignee | 知道 Assignee 表示当前事项或后续承接事项的负责人。 |
| Design PR | 知道 Design PR 用来提交设计资产，不承载 Demo 代码。 |
| Demo PR | 知道 Demo PR 用来验证设计意图，默认保持 Draft，不作为生产代码合并。 |

## 需要具备的基础能力

### 1. 读取协作上下文

协作者应能找到并阅读当前 Issue，理解背景、用户问题、期望结果和验收方向。

开始工作前，需要确认：

- 是否已有 Issue；
- Issue 是否描述了真实问题，而不是只有解决方案；
- 当前工作是否与 Issue 范围一致；
- 是否已有相关 Design PR、Demo PR 或后续实现 PR；
- 当前事项的 Assignee 是否明确。

### 2. 判断交付类型

协作者需要判断当前工作属于哪一类：

| 类型 | 说明 | 常见产物 |
| ---- | ---- | -------- |
| 设计资产维护 | 修改页面结构、组件状态、设计资源或设计系统内容。 | Design PR |
| Demo 验证 | 用可体验原型、Demo、录屏或预览说明验证关键交互、状态或用户路径。 | Demo PR |
| 文档维护 | 更新规范、模板、检查清单或工作流说明。 | Docs PR 或 Design PR |
| 后续生产实现承接 | 向开发人员说明已验证内容、未覆盖内容和后续处理事项。 | PR 描述、评论、交接说明 |

### 3. 维护设计资产

如果当前项目采用 Pencil 作为设计资产载体，协作者需要能够：

- 打开页面级 `.pen` 文件；
- 识别组件库 `.lib.pen` 与页面 `.pen` 的区别；
- 知道页面文件应优先引用组件库，而不是复制组件；
- 知道图片、字体、图标等资源需要放在约定目录中；
- 知道临时截图、废弃稿和无法追溯来源的资源不应进入仓库历史。

### 4. 使用 Git 做最小协作

协作者不需要掌握所有 Git 命令，但必须能完成最小闭环：

```text
同步目标分支
  ↓
创建工作分支
  ↓
修改文件
  ↓
检查改动范围
  ↓
提交 Commit
  ↓
推送到 origin
  ↓
创建 PR
  ↓
根据评审修改
```

提交前必须检查：

```bash
git status --short
git diff
```

### 5. 使用 AI Agent 时控制边界

让 AI Agent 协助前，必须明确：

- 当前 Issue；
- 目标分支；
- 当前 PR 类型；
- 允许修改的文件；
- 不允许修改的文件；
- 是否允许执行 Git 命令；
- 是否允许执行 `git commit`；
- 是否需要保持 Demo PR 为 Draft。

AI 生成的改动必须由人类检查后再提交。

## 准入完成标准

当协作者能够清楚说明下面问题时，可以开始参与具体 Issue：

- 当前 Issue 要解决什么问题？
- 当前改动属于 Design PR、Demo PR、文档维护，还是后续生产实现承接？
- 本次会修改哪些文件？
- 哪些文件不应该被提交？
- Demo PR 需要验证哪些路径、交互和状态？
- 提交后由谁评审？
- 后续由谁承接生产实现？

如果这些问题无法回答，说明协作上下文还不完整，应先补充 Issue 或 PR 描述。
