# 设计资产审查清单

> - 文档类型：Checklist / 设计资产入库检查
> - 稳定性：随仓库结构和工具实践微调
> - 何时阅读：提交 Design PR 前，或评审设计资产是否适合进入仓库历史时
> - 维护原则：本文检查「资产是否可以进入仓库历史」，不判断具体设计方案体验是否合理。设计表达质量检查见 [设计审查清单](./design-review-checklist.zh.md)。

## 使用范围

当 PR 涉及以下内容时，应使用本文自查：

- 页面级 `.pen` 文件；
- 组件库 `.lib.pen` 文件；
- `ui/images/*` 图片、字体、图标等资源；
- 影响 `.pen` 文件引用关系的资源；
- 设计资产命名；
- 设计系统相关文档。

如果 PR 只修改 Issue、PR 模板或普通文档，可以按实际情况选择使用。

## 本文不检查什么

本文不判断：

- 设计方案是否美观；
- 用户路径是否合理；
- 交互体验是否完整；
- 文案是否准确；
- Demo PR 验证范围。

这些内容使用 [设计审查清单](./design-review-checklist.zh.md) 检查。

## 检查清单

### 1. 文件范围

- [ ] PR 描述已说明修改了哪些 `.pen`、`.lib.pen` 或 `ui/images/*` 文件。
- [ ] 本次改动只包含当前 Issue 相关文件。
- [ ] 没有混入无关页面、无关组件或无关资源。
- [ ] 没有提交临时文件、系统文件或本地配置。
- [ ] 新增文件放在正确目录。
- [ ] 删除文件时已说明原因和影响范围。

### 2. 命名检查

- [ ] 文件名符合 `kebab-case`。
- [ ] 中文文档使用 `.zh.md`。
- [ ] 页面文件名表达页面或业务对象。
- [ ] 组件库文件名表达组件系统或产品域。
- [ ] 图片资源名能看出所属对象和状态。
- [ ] Pencil 组件根节点命名符合 `Category/ComponentName/Variant...`。
- [ ] Slot 命名符合 `Slot/<Role>`。
- [ ] 没有 `copy`、`final`、`latest`、`test` 等临时命名。

命名细则见 [设计资产命名](./design-asset-naming.zh.md)。

### 3. Pencil 文件组织

- [ ] 页面级文件只保留当前 Issue 相关稳定内容。
- [ ] 组件库文件只维护可复用组件、组件状态、变量和 Slot。
- [ ] 页面文件优先引用组件库，没有复制稳定组件。
- [ ] 组件库修改已说明影响范围。
- [ ] 节点层级能反映页面或组件结构。
- [ ] 没有无意义分组、废弃草稿或临时截图长期保留。
- [ ] 必要交互说明已在 Design PR 或 Demo PR 中表达。

Pencil 细则见 [Pencil 设计资产规范](./pencil-guidelines.zh.md)。

### 4. 资源引用

- [ ] `.pen` 文件引用的图片、字体、图标等资源已进入仓库。
- [ ] 资源位于约定目录，例如 `ui/images/*`。
- [ ] 不依赖个人本地路径。
- [ ] 删除或替换资源时，已确认没有破坏页面引用。
- [ ] 新增资源来源和用途可以说明。
- [ ] 没有提交无来源、无用途或与当前 Issue 无关的图片。

### 5. 组件库影响

如果修改 `.lib.pen`，必须检查：

- [ ] 是否影响多个页面。
- [ ] 是否新增组件、修改组件，还是只调整示例。
- [ ] 是否更新了相关状态。
- [ ] 是否影响已有 Slot。
- [ ] 是否影响变量或 Token。
- [ ] 是否需要同步更新页面级文件。
- [ ] 是否需要在 PR 中提醒评审者重点检查。

组件库改动应比页面级改动更谨慎。

### 6. Git 状态检查

提交前执行：

```bash
git status --short
git diff --stat
```

检查：

- [ ] 没有 `.DS_Store`。
- [ ] 没有 `__MACOSX/`。
- [ ] 没有临时截图或本地配置。
- [ ] 新增资源已被 `git add`。
- [ ] 删除资源是预期行为。
- [ ] `.pen` 文件和相关资源同时提交。

如果使用 `git add .`，必须再次检查暂存区：

```bash
git diff --cached --stat
```

### 7. PR 描述检查

Design PR 应说明：

- [ ] 关联 Issue。
- [ ] 设计资产改动范围。
- [ ] 修改了哪些页面或组件。
- [ ] 是否影响组件库。
- [ ] 是否新增、删除或替换资源。
- [ ] 是否说明 Demo PR 验证范围。
- [ ] 是否存在待确认点。

如果 PR 修改组件库，应额外说明：

- [ ] 影响哪些页面或组件。
- [ ] 是否需要后续同步页面文件。
- [ ] 是否需要生产实现承接。

### 8. 评审者检查

评审者应确认：

- [ ] PR 描述足以判断改动范围。
- [ ] 文件变化与 Issue 相关。
- [ ] 设计资产组织符合规范。
- [ ] 命名符合规范。
- [ ] 资源引用完整。
- [ ] 没有系统文件、临时文件或无关文件。
- [ ] 组件库修改的影响范围已说明。
- [ ] 信息不足时，已要求作者补充说明后再继续评审。
