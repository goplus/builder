# Figma 到 Pencil 迁移指南

> - 文档类型：Tool Guide / 迁移操作指南
> - 稳定性：随 Figma、Pencil 与设计工具链更新
> - 何时阅读：从 Figma 搬运设计稿、维护 Pencil 页面文件、提交 Design PR 前
> - 维护原则：本文只描述当前阶段的迁移落地方式。迁移目标不是把 Figma 图层原样复制到 Pencil，而是把设计意图整理成可维护、可复用、可审查、可交接的仓库资产。

## 定位

本文用于补充 [Pencil 设计资产规范](./pencil-guidelines.zh.md)，说明设计稿从 Figma 迁移至 Pencil 过程中常见的处理方式、引用规则和排查方法。

## 使用范围

本文用于处理：

- 从 Figma 迁移页面、组件和资源到 Pencil；
- 整理迁移后的页面结构、组件引用、变量和资源；
- 排查迁移后的尺寸、引用、图片和 Icon 问题；
- 补充 Design PR 中的迁移说明和自查结果。

## 本文不处理什么

本文关注「怎么迁移」。它不替代：

- [设计资产命名](./design-asset-naming.zh.md)：负责定义文件、组件、节点和资源如何命名；
- [设计审查清单](./design-review-checklist.zh.md)：负责检查设计表达质量；
- [设计资产审查清单](./design-asset-review-checklist.zh.md)：负责检查资产是否适合进入仓库历史。

Figma 是设计意图的来源之一。Pencil 是当前阶段采用 Pencil 的项目把设计资产纳入仓库历史的一种载体，XBuilder 是主要试验场。迁移不是简单复制，而是一次资产整理。

## 迁移目标

从 Figma 到 Pencil 的迁移，应达成四个目标：

1. 设计意图被保留。
   - 页面结构、视觉层级、关键状态和交互关系应与 Figma 方案一致。

2. 资产结构可维护。
   - 页面和组件应使用结构化布局，避免大量无意义分组、临时图层和绝对定位。

3. 组件和变量可复用。
   - 优先使用现有组件库、设计变量、Slot 和资源目录，不重复创建局部组件或临时数值。

4. 后续协作可审查、可交接。
   - 其他协作者拉取仓库后，应能打开文件、理解结构、定位资源，并通过 PR 描述知道迁移范围和待确认点。

## 操作前准备

开始迁移前，先确认以下信息：

- 当前迁移对应的 Issue；
- 迁移范围是页面、组件、资源，还是三者都有；
- 目标文件应放在页面目录、组件库目录还是资源目录；
- Figma 中哪一个 Frame 或版本是本次迁移依据；
- 当前组件库中是否已有可复用组件；
- 当前设计变量中是否已有对应颜色、间距、圆角和阴影；
- 本次迁移是否需要新增组件、变量或图片资源。

不要把 Figma 中的探索过程、废弃方案、标注截图和临时参考图直接带入 Pencil。进入仓库历史的应该是整理后的协作资产，而不是设计过程的中间残留。

## 核心原则

### 1. 保留设计意图，不保留未整理图层

Figma 文件中可能存在大量临时分组、截图、隐藏图层、标注、废弃方案、探索稿或复制出来的中间状态。

迁移到 Pencil 时，应保留：

- 页面主结构；
- 关键视觉层级；
- 真实需要维护的组件；
- 当前 Issue 需要覆盖的状态；
- 后续实现需要理解的资源和布局关系。

应删除或避免迁移：

- 隐藏图层；
- 废弃方案；
- 无关探索稿；
- 临时截图；
- 标注线、说明框、测量辅助图；
- 与当前 Issue 无关的页面；
- 复制后未整理的 Frame 或 Group。

### 2. 优先使用组件库，不在页面中复制组件结构

页面文件应引用组件库中的组件，而不是在页面中重新创建一套局部组件。

迁移时的判断顺序：

1. 组件库中是否已经有同类组件；
2. 现有组件是否可以通过状态、Slot 或属性满足当前页面；
3. 如果不能满足，是否需要扩展现有组件；
4. 如果是全新模式，是否需要新增组件并进入组件库；
5. 如果只是当前页面一次性的局部结构，是否可以留在页面文件中。

不要因为从 Figma 复制成本低，就在页面级文件里维护一份和组件库重复的结构。

### 3. 使用设计变量，不随手写临时数值

从 Figma 搬运设计稿时，应优先使用 Pencil 中已经建立的设计变量，而不是直接输入临时数值。

设计变量包括但不限于：

- Color；
- Typography；
- Padding；
- Gap；
- Radius；
- Shadow。

处理方式：

| Figma 中的内容 | Pencil 中的处理方式 |
| ---- | ---- |
| 颜色 | 映射到已有 Color 变量；缺失时先确认是否为新的设计决策。 |
| 字号 / 字重 / 行高 | 优先匹配已有 Typography 规则；不要创建无法复用的局部样式。 |
| Padding / Gap | 优先匹配组件库中已命名的间距变量。 |
| Radius | 优先匹配已有圆角变量。 |
| Shadow | 优先匹配已有阴影变量。 |
| 特殊数值 | 在 PR 中说明原因，确认后再沉淀为变量。 |

如果现有变量无法满足设计需求，不要直接新增临时变量。应先判断这是否是新的设计决策，必要时在 PR 中写明：

- 为什么现有变量无法覆盖；
- 新变量会影响哪些组件或页面；
- 是否需要同步到设计系统。

### 4. 用结构化布局表达页面关系

Pencil 文件应表达真实页面结构，而不是把视觉结果固定在一堆绝对坐标上。

建议：

- 页面结构使用 Auto Layout；
- 组件内部使用 Auto Layout；
- 有内容承载关系的容器优先使用 Auto Layout；
- 布局调整优先通过 Gap、Padding、Align 和适应模式完成；
- 减少纯 Frame 和无意义 Group；
- 避免用绝对定位强行还原视觉结果。

只有在浮层、自由拖拽、画布对象、装饰元素等确实需要自由定位的场景中，才考虑保留绝对定位。

## 操作规则

### 尺寸与适应模式

Figma 内容迁移到 Pencil 后，可能出现尺寸变化、布局错位、容器异常拉伸或内容溢出等问题。

排查时不要只改宽高，应按顺序检查：

1. Figma 中原始 Frame 的宽高；
2. Pencil 中根容器的宽高；
3. 父容器的适应模式；
4. 子节点的适应模式；
5. Auto Layout 的方向、Gap、Padding 和对齐方式；
6. 是否存在意外旋转或小数偏移。

适应模式建议：

| 场景 | 建议处理 |
| ---- | ---- |
| 页面根容器 | 通常保留固定宽高，便于和 Figma 原稿对齐。 |
| 横向或纵向容器 | 优先使用 Auto Layout，并通过 Gap / Padding 管理间距。 |
| 内容随文本或子项变化 | 通常使用 Hug。 |
| 内容需要占满父容器剩余空间 | 通常使用 Fill。 |
| 固定尺寸图标、头像、按钮高度 | 保留固定尺寸。 |

注意：原始草稿中的「File 或 Hug」应校正为「Fill 或 Hug」。如 Pencil 当前版本的界面命名发生变化，以工具面板实际显示为准。

页面还原应以 Figma 中的实际尺寸和布局关系为准，不应完全依赖迁移后的默认结果。

### 命名规范

组件、页面、节点和资源应遵循 [设计资产命名](./design-asset-naming.zh.md)。

#### 页面文件命名

页面级文件建议采用：

```text
<business-module>-<page-or-scenario>.pen
```

示例：

```text
community-login.pen
community-home.pen
community-profile.pen
editor-asset-panel.pen
editor-sprite-item.pen
```

命名时应保持当前仓库已有风格一致，不混用大小写、空格、下划线和中文临时命名。

#### 组件命名

新增组件前应先确认：

1. 所属组件分类；
2. 是否已有同类组件；
3. 是否应放入组件库文件；
4. 是否需要 Slot；
5. 是否包含多个状态；
6. 命名方式是否与同类组件一致。

不要出现以下命名：

```text
组件1
新按钮
copy
final
latest
临时
未命名
```

#### 节点命名

页面内部节点应能帮助评审者理解结构。

建议命名表达节点职责，例如：

```text
Header
Sidebar
AssetPanel
SpriteList
SpriteItem
Toolbar
PreviewArea
EmptyState
LoadingState
ErrorState
```

不要保留 Figma 自动生成或复制产生的无意义名称，例如：

```text
Frame 1234
Group 56
Rectangle 8 Copy 3
```

### Slot 占位符使用

组件库中的组件应通过 Slot 占位符实现复用和替换。

#### 创建 Slot 时

建议：

1. 将同类型组件放置在同一分组中；
2. 创建对应占位符；
3. 使用清晰命名表达 Slot 角色；
4. 通过 Slot 机制绑定组件；
5. 占位符自身使用 Auto Layout；
6. 根据内容关系设置 Hug、Fill 或固定尺寸。

Slot 命名建议：

```text
Slot/Icon
Slot/Label
Slot/Action
Slot/Content
Slot/Preview
```

#### 使用 Slot 时

在页面文件中使用组件库 Slot 的推荐方式：

1. 从组件库复制对应占位符或组件实例；
2. 粘贴到目标页面文件；
3. 如出现引用弹窗，选择 `Yes`；
4. 确认页面中生成的是子级实例，而不是复制出来的独立组件；
5. 在 Slot 中选择对应组件或内容；
6. 检查实例是否能跟随父级组件更新。

页面结构应尽量采用「占位符 + 组件实例」完成搭建，而不是拆开组件内部结构重新拼装。

### 跨文件引用组件库

页面级文件应引用组件库中的组件，而不是重新创建组件来源。

首次在页面文件中粘贴组件库占位符时：

1. 打开组件库文件；
2. 复制组件库中的占位符或组件实例；
3. 粘贴至页面文件；
4. 出现引用弹窗时选择 `Yes`；
5. 确认页面中生成对应子级实例；
6. 保存页面文件；
7. 提交页面文件及必要的资源文件。

组件维护关系应保持为：

```text
组件库文件：维护父级组件、组件状态、Slot 和变量。
页面级文件：使用子级实例，表达具体业务页面和状态。
```

除组件库文件外，不应维护独立组件来源。若必须临时放置局部结构，应在 PR 中说明原因和后续处理方式。

### 图片与 Icon 使用

#### 图片资源

Figma 中的背景图、插画、手绘图、位图类素材，应导出后放入仓库资源目录。

当前建议目录：

```text
ui/images/
```

要求：

- 图片应使用稳定命名；
- 图片应随 `.pen` 文件一起提交；
- 不引用个人本地路径；
- 不引用未提交资源；
- 不把临时截图作为正式资源长期保留；
- 如存在压缩、裁剪或导出倍率要求，应在 PR 中说明。

位图资源通常可导出为 PNG。若后续项目统一采用 SVG、WebP 或其他格式，应以仓库规范为准。

#### Icon 资源

当前项目中的 Icon 采用字体方式维护，组件库文件中维护对应 Icon 对照表。

使用 Icon 时：

1. 从组件库复制对应 Icon 占位符或实例；
2. 放置到目标位置；
3. 调整字体大小；
4. 调整颜色变量；
5. 检查字体是否正确；
6. 检查页面中是否显示为预期 Icon。

不要把组件库中已有的 Icon 单独导出为图片再导入页面。这样会造成资源重复，也会破坏后续统一维护能力。

## 常见问题排查

### 1. 宽高异常或页面错位

优先检查：

- Figma 原始宽高；
- Pencil 根容器宽高；
- 父子容器适应模式；
- Auto Layout 方向；
- Gap、Padding 和 Align；
- 是否存在隐藏内容撑开容器。

不要直接通过强行修改尺寸来掩盖结构问题。

### 2. 出现 0.01 偏移或对齐异常

此类问题可能不是尺寸问题，而是组件或图层被意外旋转导致。

排查方式：

1. 检查异常组件；
2. 检查组件内部图层；
3. 查看 Rotation 属性；
4. 找出存在角度偏移的节点；
5. 将 Rotation 恢复为 `0`；
6. 再重新检查宽高和对齐。

发现异常时应逐层排查，不应直接通过修改尺寸强行修正结果。

### 3. 组件实例没有跟随组件库更新

优先检查：

- 页面中使用的是不是组件库实例；
- 首次跨文件粘贴时是否选择了 `Yes`；
- 是否把组件拆开后变成了普通图层；
- 是否在页面文件中重新创建了独立组件来源；
- 组件库文件是否已经保存并提交。

如果页面中已经出现重复组件来源，应回到组件库重新引用，不建议继续在页面内维护副本。

### 4. 图片丢失或他人无法打开

优先检查：

- 图片是否放在 `ui/images/`；
- 图片是否已被 `git add`；
- `.pen` 文件是否引用了仓库内资源；
- 是否引用了个人电脑本地路径；
- 是否只提交了 `.pen` 文件但漏提交图片。

新增图片资源时，不要使用 `git commit -am` 代替 `git add`。`git commit -am` 不会提交新文件。

### 5. Icon 显示异常

优先检查：

- 是否使用了组件库中的 Icon 占位符；
- 字体是否为项目约定的 Icon 字体；
- 字号是否正确；
- 颜色是否使用变量；
- 是否误把 Icon 当成普通文本或图片处理。

## 提交前检查

### Design PR 中需要说明什么

如果本次 PR 包含 Figma 到 Pencil 的迁移，PR 描述中应说明：

- 对应 Issue；
- Figma 来源范围；
- 迁移了哪些页面或组件；
- 新增或修改了哪些 `.pen` / `.lib.pen` 文件；
- 新增了哪些图片资源；
- 是否新增或修改设计变量；
- 是否新增组件或 Slot；
- 与 Figma 原稿相比是否有整理、删减或调整；
- 是否存在待确认点；
- 后续 Demo PR 的验证范围。

可复制描述：

```md
## Figma 到 Pencil 迁移说明

- Figma 来源：
- 迁移范围：
- 修改文件：
- 新增资源：
- 新增 / 修改组件：
- 新增 / 修改变量：
- 与 Figma 原稿的差异：
- 待确认点：
- Demo PR 验证范围：
```

### 迁移后自查清单

提交前，应完成以下检查：

### 设计变量

- [ ] 颜色使用已有 Color 变量。
- [ ] 字号、字重、行高匹配已有 Typography 规则。
- [ ] Padding、Gap、Radius、Shadow 优先使用已有变量。
- [ ] 新增变量已在 PR 中说明原因。

### 布局结构

- [ ] 页面结构使用 Auto Layout 表达主要关系。
- [ ] 组件内部使用 Auto Layout。
- [ ] Gap、Padding 和 Align 设置合理。
- [ ] 没有用大量绝对定位保存最终视觉结果。
- [ ] 根容器尺寸与 Figma 依据稿一致。

### 组件与 Slot

- [ ] 页面优先引用组件库，而不是复制组件内部结构。
- [ ] Slot 命名清晰，能表达替换区域职责。
- [ ] 页面中使用的是组件库实例。
- [ ] 组件库更新后，页面实例可以同步。

### 命名与资源

- [ ] 页面文件、组件、节点和图片命名符合规范。
- [ ] 删除了 Figma 复制产生的无意义名称。
- [ ] 图片资源已放入 `ui/images/`。
- [ ] 新增资源已提交到 Git。
- [ ] 没有引用个人本地路径。

### 异常排查

- [ ] 没有明显宽高异常。
- [ ] 没有异常旋转。
- [ ] 没有 0.01 偏移导致的对齐问题。
- [ ] Icon 显示正常。
- [ ] 他人拉取仓库后可以打开并查看相关页面。

## 结论

Figma 到 Pencil 的迁移，不是把一个工具中的图层搬到另一个工具中，而是把设计稿整理为工程协作资产。

好的迁移结果应该同时满足：

- 视觉与设计意图可靠；
- 结构清晰；
- 组件可复用；
- 变量可维护；
- 资源可追溯；
- PR 可审查；
- 后续实现可交接。
