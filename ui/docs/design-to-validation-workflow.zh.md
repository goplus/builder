# 设计到体验验证工作流

> 文档类型：Workflow / 主流程
> 稳定性：长期稳定
> 适用对象：产品设计工程化协作者、体验验证材料作者、Demo 实现者、Reviewer、后续生产实现承接人
> 何时阅读：开始一个 Issue、拆分 Design PR 与体验验证材料、判断是否需要 Demo PR 时
> 维护原则：本文定义阶段、对象、责任和边界；不定义 Pencil 文件细节、不定义生产代码合并规则。

## 定义

设计到体验验证工作流，是一套适用于产品设计、UI 设计和体验验证项目的协作流程。它用于在后续生产实现之前，整理设计资产、验证设计意图、推动团队形成产品设计共识，并保留交接上下文。XBuilder 是当前阶段的主要试验场。

它的主链路是：

```text
Issue
  ↓
Design PR
  ↓
Demo PR
  ↓
后续生产实现
```

这条链路的重点不是让所有需求都变复杂，而是让设计在进入生产实现之前，先被整理、审查、体验验证，并形成可交接的设计共识。

## 工作流边界

本文适用于：

- 需要修改设计资产的体验改动；
- 需要通过可体验原型或 Demo 验证关键交互的设计；
- 涉及页面状态、组件状态、动效、浮层、拖拽、AI 生成过程等静态稿难以表达的场景；
- 需要把设计意图交接给前端生产实现的任务。

本文不适用于：

- 只修改生产代码、不涉及设计资产的工程任务；
- 只修改后端逻辑、接口或配置的任务；
- 单纯探索性草稿；
- 未准备进入仓库历史的临时设计讨论。

## 核心对象

| 对象 | 作用 | 是否承载文件改动 | 合并语义 |
| ---- | ---- | ---------------- | -------- |
| Issue | 记录问题、背景、目标和验收方向。 | 否 | 工作上下文起点。 |
| Design PR | 提交设计资产，说明设计意图和变更范围。 | 是 | 可合并，合并前只保留应进入仓库历史的设计资产。 |
| 体验验证材料 | 提供可体验原型、Demo、录屏、预览入口或验证说明。 | 视情况而定 | 用于验证设计意图和形成团队共识。 |
| Demo PR | 提供 Demo 代码、预览入口和验证说明。 | 是 | 默认保持 Draft，不作为生产代码合并。 |
| 后续生产实现 | 承接已验证的设计意图，完成生产代码。 | 是 | 按生产研发流程处理，不由本文定义。 |

同一个需求应通过同一个 Issue 串联 Design PR、Demo PR 和后续生产实现，避免设计资产、验证结论和代码实现脱节。

## 阶段一：Issue

Issue 是工作起点。开始设计前，必须先确认当前工作对应哪个 Issue。

一个合格的 Issue 应包含：

- 背景：为什么需要做这件事；
- 问题：当前用户体验或产品表达有什么断点；
- 目标：希望改善什么；
- 期望结果：完成后用户能获得什么；
- 验收方向：如何判断这件事可以进入下一阶段；
- 关联材料：截图、录屏、设计稿、讨论链接、现有页面入口等。

如果 Issue 只写了「改一下 UI」「优化一下体验」这类描述，应先补充上下文，再进入 Design PR。

## 阶段二：Design PR

Design PR 用来提交设计资产。当前阶段，设计资产主要包括：

- 页面级 `.pen` 文件；
- 组件库 `.lib.pen` 文件；
- `ui/images/*` 中被设计资产引用的图片、字体、图标等资源；
- 与设计资产维护直接相关的文档。

Design PR 应说明：

- 关联 Issue；
- 本次设计要解决什么问题；
- 修改了哪些设计资产；
- 关键页面、组件、状态或交互是什么；
- 是否影响设计系统；
- Demo PR 需要验证哪些路径、交互和状态；
- 是否存在待确认点。

Design PR 的评审重点是：

- 设计表达是否完整；
- 状态是否覆盖；
- 组件和设计系统是否一致；
- 资产是否适合进入仓库历史；
- 后续体验验证或生产实现是否能理解设计意图。

Design PR 可以合并，但合并前应该清理临时内容，只保留应该进入仓库历史的设计资产。

## 阶段三：Demo PR

Demo PR 是体验验证的固定承载方式。

Design PR 提交设计依据后，应通过 Demo PR 提供可体验验证材料。Demo PR 可以包含前端 Demo 代码、预览环境、截图、录屏或验证说明。

Demo PR 应说明：

- 关联 Issue 和 Design PR；
- Demo 入口；
- 验证了哪些用户路径；
- 验证了哪些交互和状态；
- 哪些内容只是 Demo 表达；
- 哪些内容需要后续生产实现承接。

Demo PR 默认保持 Draft 状态。它的目的不是生产合并，而是体验验证、团队讨论、共识形成和后续交接。

## 阶段四：后续生产实现

后续生产实现基于 Issue、Design PR、Demo PR、体验验证结论和预览结果继续完成。

生产实现需要处理：

- 真实数据；
- 权限；
- 状态管理；
- 异常兜底；
- 性能；
- 工程结构；
- 测试；
- 上线策略。

这些内容不属于 Design PR 或 Demo PR 的责任范围。但 Design PR、体验验证材料和 Demo PR 必须提供足够清晰的设计依据，降低后续实现中的猜测成本。

## Demo PR 的验证范围

Demo PR 是固定环节，但验证范围应与设计风险匹配。

需要重点验证的情况：

- 存在关键交互路径；
- 静态稿无法说明状态变化；
- 涉及拖拽、展开、收起、浮层、面板避让；
- 涉及 AI 生成中、失败、重试、取消、回滚；
- 涉及多状态切换；
- 涉及动效、过渡或空间关系变化；
- 涉及编辑器、素材面板、地图模式等复杂操作区；
- 开发人员容易根据静态稿产生误解。

可以轻量验证的情况：

- 单纯文案修改；
- 图标替换；
- 颜色、间距、圆角等小范围视觉调整；
- 静态页面布局微调；
- 不影响交互路径的设计资产整理；
- 文档或模板维护。

判断标准是：Demo PR 需要提供多少体验证据，才能支持团队形成设计共识并交接给后续生产实现。

## Design PR 与 Demo PR 的拆分原则

Design PR 和 Demo PR 不应该默认混在一起。

建议拆分的情况：

- 设计资产需要合并进 `ui`，Demo 代码不应合并；
- Demo 只是验证表达，不具备生产质量；
- 评审参与者需要分别审查设计资产和体验验证材料；
- 同一 Issue 后续还需要开发人员承接生产实现。

可以合并在同一个 PR 中的情况应非常谨慎，只适用于极小范围、无 Demo 代码、无复杂状态、无生产误合并风险的文档或设计资产更新。

## 分支建议

建议按 Issue 创建独立工作分支。

Design PR 分支：

```text
design/issue-1234-short-name
```

Demo PR 分支：

```text
demo/issue-1234-short-name
```

文档更新分支：

```text
docs/issue-1234-short-name
```

分支名中保留 Issue 编号，便于追溯。

## 交接说明

Design PR、体验验证材料或 Demo PR 进入后续生产实现前，应在 PR 描述或 Issue 评论中保留交接说明。

建议至少包含：

- 当前体验验证的结论；
- 后续实现必须保留的交互和状态；
- Demo 中哪些内容不能直接作为生产实现；
- 尚未覆盖的异常情况；
- 需要开发继续确认的技术点；
- 生产实现 Assignee 或后续承接 Issue。

## 评审关注点

评审参与者应分别检查 Design PR 和 Demo PR。

Design PR 关注：

- 设计目标是否与 Issue 一致；
- 页面结构是否清楚；
- 状态是否完整；
- 组件复用是否合理；
- 设计系统是否一致；
- 文件和资源是否适合入库。

Demo PR 关注：

- 预览入口是否可用；
- Demo 是否覆盖关键路径；
- 交互是否符合设计意图；
- PR 是否明确非生产代码边界；
- 后续生产实现需要承接的内容是否清楚；
- PR 是否保持 Draft。

## 常见问题处理

### 发现 Issue 描述不足

先补充 Issue，再继续设计。不要用 PR 描述替代 Issue 背景。

### Design PR 中混入 Demo 代码

拆分 PR。设计资产进入 Design PR，Demo 代码进入 Demo PR。

### Demo PR 中出现可合并的设计资产改动

先把设计资产拆到 Design PR，评审通过后再让 Demo PR 基于已确认的设计资产继续验证。

### Demo 结果推翻原设计

回到 Design PR 更新设计资产，并在 Issue 中记录结论。不要只在 Demo PR 中改代码而不更新设计依据。

### 后续实现发现设计未覆盖异常状态

补充 Issue 或在原 Issue 中追加待办，再按需要更新 Design PR 或 Demo PR。
